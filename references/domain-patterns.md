# 领域动态生成规则

> 上帝不能固定生成"学习教练团队"。它必须先识别领域类型，再动态生成不同组织结构、不同 shared/ 文件、不同知识库。

## 核心原则

**团队结构不是固定模板，而是根据领域类型动态生成。**

上帝在 Phase 0 判断需求类型后，还要进一步判断领域类型，然后按该领域的模式生成对应的：
1. 角色分工
2. 共享知识库结构
3. 成员专属知识库结构
4. 用户记忆文件
5. 复盘周期和方式

---

## 一、领域类型识别

| 领域类型 | 识别信号 | 核心特征 |
|---------|---------|---------|
| **考试学习型** | 备考、考试、提分、课程 | 有明确分数目标和截止日期，需要评分反馈 |
| **健身健康型** | 健身、减肥、训练、饮食 | 有体能指标，需要训练计划和恢复管理 |
| **宠物照护型** | 宠物、猫、狗、养护 | 有生命体观察，需要健康记录和异常预警 |
| **选购决策型** | 买、选、对比、预算 | 有决策过程，需要需求诊断和对比分析 |
| **产品执行型** | 产品、需求、迭代、用户 | 有产品生命周期，需要决策日志和路线图 |
| **内容创作型** | 写作、视频、运营、涨粉 | 有内容产出，需要数据反馈和风格记忆 |
| **研究分析型** | 研究、分析、论文、调研 | 有知识积累，需要来源索引和观点演化 |
| **技能习得型** | 学、练、掌握（非考试） | 有技能进阶，需要里程碑和熟练度评估 |

## 二、各领域的 shared/ 文件模板

### 考试学习型

```
shared/
├── user-profile.md       # 目标分数、考试日期、当前水平
├── progress.md           # 练习记录、分数趋势
├── weak-points.md        # 弱点状态追踪
├── session-log.md        # 会话摘要
├── weekly-reviews.md     # 周复盘
├── monthly-summary.md    # 月度总结
└── archive/              # 历史归档
```

### 健身健康型

```
shared/
├── user-profile.md       # 身体数据、目标、运动经验
├── training-log.md       # 训练记录（日期、动作、组数、重量）
├── body-metrics.md       # 体重、体脂、围度趋势
├── nutrition-log.md      # 饮食记录
├── injury-risk.md        # 伤病风险和恢复状态
├── weekly-reviews.md     # 周复盘
├── monthly-summary.md    # 月度总结
└── archive/
```

### 宠物照护型

```
shared/
├── pet-profile.md        # 品种、年龄、体重、疫苗、绝育
├── health-log.md         # 健康观察记录
├── feeding-log.md        # 饮食记录
├── vet-visits.md         # 就医记录
├── alerts.md             # 异常症状和预警
├── weekly-reviews.md     # 周观察总结
└── archive/
```

### 选购决策型

```

├── preferences.md        # 用户需求、预算、偏好
├── comparison-table.md   # 候选对比表
├── decision-log.md       # 决策过程记录
├── purchase-record.md    # 已购买记录
├── maintenance-log.md    # 维护/售后记录
└── archive/
```

### 产品执行型

```
├── decision-log.md       # 产品决策日志
├── experiment-log.md     # 实验/AB测试记录
├── user-feedback.md      # 用户反馈汇总
├── roadmap.md            # 产品路线图
├── weekly-reviews.md     # 周站会总结
├── monthly-summary.md    # 月度复盘
└── archive/
```

### 内容创作型

```

├── creator-profile.md    # 定位、风格、目标受众
├── content-log.md        # 发布记录（标题、数据、反馈）
├── topic-bank.md         # 选题库
├── style-memory.md       # 风格记忆（爆款模式、用户偏好）
├── weekly-reviews.md     # 周数据复盘
├── monthly-summary.md    # 月度总结
└── archive/
```

### 研究分析型

```
shared/
├── research-brief.md     # 研究问题、范围、方法论

├── findings-log.md       # 发现记录
```

### 技能习得型

```
├── weekly-reviews.md     # 周复盘
├── monthly-summary.md    # 月度总结

```

## 三、各领域的角色设计模式

### 考试学习型
```
协调器 + 各科教练 + 规划师
（可选：错题复盘官、心理教练）
```

### 健身健康型
```
协调器 + 训练教练 + 营养师 + 恢复顾问
（可选：运动医学顾问）
```

### 宠物照护型
```
协调器 + 日常照护顾问 + 健康观察员 + 营养顾问
（可选：行为训练师、品种专家）
```

### 选购决策型
```
协调器 + 需求诊断师 + 市场分析师 + 对比评测员
（可选：售后顾问）
```

### 产品执行型
```
协调器 + 产品经理 + 用户研究员 + 技术架构师
（可选：数据分析师、增长专家）
```

### 内容创作型
```
协调器 + 选题策划 + 内容创作者 + 数据分析师
（可选：视觉设计顾问、分发策略师）
```

### 研究分析型
```
协调器 + 信息采集员 + 分析师 + 综合报告员
（可选：批判性审查员）
```

## 四、领域识别规则（Phase 0 使用）

```
Step 1: 从用户需求中提取关键词
Step 2: 匹配领域类型（见上表"识别信号"）
Step 3: 如果匹配到 → 使用该领域的默认结构
Step 4: 如果不匹配 → 询问用户或从需求推断最接近的类型
Step 5: 展示推荐结构，用户确认后开始生成
```

**注意**：领域类型只是起点。用户可以要求增删角色或文件。上帝的能力是动态设计，不是套模板。

## 五、通用记忆文件（所有领域必须有）

无论什么领域，以下文件是通用的：

| 文件 | 用途 | 必须/推荐 |
|------|------|---------|
| `[主体]-profile.md` | 核心对象的档案（用户/宠物/产品/项目） | 必须 |
| `[核心动作]-log.md` | 主要活动的记录 | 必须 |
| `weekly-reviews.md` | 周复盘 | 推荐 |
| `archive/` | 历史归档 | 推荐 |

弱点追踪 (`weak-points.md`) 只在有"能力提升"属性的领域使用（考试、技能、健身）。

决策日志 (`decision-log.md`) 只在有"决策"属性的领域使用（选购、产品、研究）。

## 六、复盘周期规则

| 领域类型 | 周复盘 | 月总结 | 归档周期 |
|---------|--------|--------|---------|
| 考试学习型 | 每周 | 每月 | session-log 7 天 |
| 健身健康型 | 每周 | 每月 | training-log 14 天 |
| 宠物照护型 | 每周 | 每月 | health-log 30 天 |
| 选购决策型 | 不需要 | 不需要 | 购买后归档 |
| 产品执行型 | 每周 | 每月/每迭代 | decision-log 30 天 |
| 内容创作型 | 每周 | 每月 | content-log 30 天 |
| 研究分析型 | 每周或按阶段 | 按阶段 | 不主动归档 |
