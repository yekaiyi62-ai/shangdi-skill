# 领域设计维度参考

> 上帝不套模板。这份文件是「思考工具」，不是「配置表」。
> 
> 用它来问对问题，从问题的答案推演出这个用户专属的结构——
> 同样是「考试备考」，雅思团队和高考团队生成出来应该完全不同。

---

## 核心原则

**团队结构是推演出来的，不是选出来的。**

上帝在 Phase 0 做的不是「匹配领域 → 套结构」，而是：

```
识别领域类型
    ↓
用该领域的设计问题清单提问
    ↓
从用户的具体答案推演出专属结构
    ↓
展示结构草案给用户确认
    ↓
用户调整后开始生成
```

领域类型只是提问的起点，不是输出的终点。

---

## 一、领域类型识别

| 领域类型 | 识别信号 | 核心特征 |
|---------|---------|---------|
| **考试学习型** | 备考、考试、提分、攻克某科 | 有明确目标分数和截止日期，评分是客观的 |
| **健身健康型** | 健身、减肥、增肌、体能训练 | 有可测量的身体指标，存在恢复和伤病管理 |
| **宠物照护型** | 宠物、猫、狗、鸟、养护 | 照护对象无法表达，需要观察和记录模式 |
| **选购决策型** | 买、选、对比、预算、值不值 | 有一个决策终点，需求探索 → 对比 → 防坑 |
| **产品执行型** | 产品、需求、迭代、用户增长 | 有产品生命周期，决策多且需要留痕 |
| **内容创作型** | 写作、视频、运营、涨粉、变现 | 有内容产出周期，需要数据反馈闭环 |
| **研究分析型** | 研究、论文、调研、分析报告 | 有知识积累过程，来源管理和观点演化是核心 |
| **技能习得型** | 学、练、掌握（非考试导向）| 有里程碑和熟练度，主观进度多于客观评分 |

> 不确定时，多个类型可以并存。一个「准备 CFA + 同时健身」的用户可以生成两个独立团队，或一个跨领域的协调器。

---

## 二、各领域的核心设计问题

**上帝在 Phase 0B 用这些问题和用户对话，从答案推演团队结构。**

### 考试学习型

```
1. 这个考试有几个科目/维度？评分结构是什么？
   → 决定教练数量和分工方式
   → 雅思 4 科 → 4 个教练；CFA 1 级 10 个科目 → 可能需要分组管理

2. 距考试还有多长时间？
   → 决定记忆文件的时间粒度
   → 3 个月以上 → 需要月度总结；1 个月以内 → 日计划比周计划更重要

3. 各科是否有客观的评分标准？
   → 决定弱点追踪的粒度
   → 雅思有 4 维度评分 → 弱点可以细化到维度级；数学考试只有对错 → 弱点细化到题型

4. 有没有模拟考机制？
   → 决定是否需要 mock-test-log 和节奏管理
   → 有 → 安排定期模拟考，追踪全科预估分；无 → 单科练习为主

5. 用户的最弱项是哪里？距目标差最多？
   → 决定哪个教练最重要、资源分配优先级

6. 备考是否有伴侣/焦虑问题？
   → 决定是否需要心理支持角色（对长期备考尤其重要）
```

**推演示例**：

| 同为考试学习型 | 雅思备考（2 个月）| 高考备考（1 年）| CFA 1 级（6 个月）|
|-------------|----------------|---------------|-----------------|
| 教练分工 | 4 科教练各一 | 12 科教练（可分组）| 10 科模块化 |
| 时间粒度 | 周计划 + 日推荐 | 月计划 + 周计划 + 日计划 | 按备考阶段划分 |
| 模拟考 | 定期全科模拟 | 月度模拟考 | 官方 Mock Exam |
| 记忆文件 | session-log（7天）+ weekly | 月报 + 重要节点 | 章节进度 + 错题库 |
| 特有文件 | 无 | 志愿填报记录 | 计算器练习记录 |

### 健身健康型

```
1. 目标是减脂、增肌、还是体能提升？
   → 决定训练师的侧重和营养师的方向

2. 是否有伤病或身体限制？
   → 决定是否需要恢复顾问和运动医学角色

3. 是否追踪饮食？
   → 决定是否需要营养日志（不是所有人都愿意记食物）

4. 训练环境是健身房、家、还是户外？
   → 影响训练师给出的方案类型

5. 目标有时间节点吗（备赛、婚礼、夏天）？
   → 影响计划的节奏和强度

6. 是否需要体测数据追踪（体重/体脂/围度）？
   → 决定 body-metrics 的记录频率
```

---

### 宠物照护型

```
1. 是哪种宠物？品种？年龄？
   → 不同品种有不同健康风险（比如英短和心脏病、金毛和髋关节）

2. 目前是否有健康问题？
   → 决定是否需要「症状追踪」和「药物记录」

3. 饮食是自己做还是成品粮？
   → 决定营养顾问的介入深度

4. 是否有行为问题需要改善（乱叫/攻击/焦虑）？
   → 决定是否需要行为训练师角色

5. 主人的时间投入能力？
   → 决定照护计划的密度
```

---

### 选购决策型

```
1. 买什么？价格区间？
   → 决定对比的维度和防坑的重点

2. 决策时间有多紧迫？
   → 紧急 → 快速 Top3 推荐；充裕 → 深度对比分析

3. 是否已有倾向品牌/产品？
   → 决定是「验证」还是「探索」模式

4. 最担心什么（质量/性价比/售后/上手难度）？
   → 决定对比评测员的权重体系

5. 买完之后还需要跟进吗（售后/维护）？
   → 决定是否需要购后记录
```

---

### 产品执行型

```
1. 处于产品的哪个阶段（0→1 / 增长 / 成熟 / 转型）？
   → 决定团队角色的重心

2. 团队规模？
   → 独立开发者 → 上帝扮演多个角色；完整团队 → 上帝辅助某个角色

3. 主要痛点是什么（需求混乱/执行低效/数据看不懂）？
   → 决定哪个角色最重要

4. 是否需要对外呈现（给投资人/董事会）？
   → 决定是否需要报告生成角色

5. 迭代节奏是多长？
   → 决定 sprint-log 的时间颗粒度
```

---

### 内容创作型

```
1. 做什么平台？（文字/视频/播客/图文）
   → 决定内容形式和对应角色

2. 当前阶段（0 粉起步 / 中期增长 / 变现优化）？
   → 决定是「选题探索」还是「数据优化」模式

3. 更新频率目标？
   → 决定内容日历的密度和规划窗口

4. 是否有合作/团队？
   → 决定流程管理的复杂度

5. 数据关注哪些指标（播放/转化/粉丝/GMV）？
   → 决定数据分析师的关注维度
```

---

### 研究分析型

```
1. 研究目的（学术论文 / 商业报告 / 个人探索）？
   → 决定严谨程度和引用格式要求

2. 已有多少背景知识？
   → 决定「文献综述」的起点深度

3. 研究周期多长？
   → 决定进度管理的粒度

4. 最终产出是什么形式（报告/幻灯片/论文/内部文档）？
   → 决定是否需要专门的写作/排版角色

5. 是否需要协作/审阅？
   → 决定是否需要批判性审查员
```

---

### 技能习得型

```
1. 具体是什么技能？有没有行业标准或段位体系？
   → 决定「里程碑」的定义方式

2. 是否有教材/课程在跟？
   → 决定上帝是「补充」还是「主导」

3. 最终应用场景是什么？
   → 决定练习的侧重（应试/实战/展示）

4. 学到什么程度算「掌握」？
   → 决定熟练度评估标准

5. 有没有练习伙伴或反馈来源？
   → 决定上帝的反馈频率
```

## 三、通用记忆文件原则

**不预设文件，从三个问题推演：**

```
1. 这个领域的「主要活动」是什么？
   → 活动日志：[活动]-log.md
   → 例：写作练习 → writing-log / 训练 → training-log / 浏览文献 → reading-log

2. 这个领域的「核心对象」是什么？
   → 对象档案：[对象]-profile.md
   → 例：用户 → user-profile / 宠物 → pet-profile / 产品 → product-brief

3. 这个领域需要追踪「弱点/风险/待改进」吗？
   → 是 → weak-points.md 或 risk-log.md（按领域语义选命名）
   → 否（如选购决策）→ 不生成
```

**以下三个文件所有领域通用：**

| 文件 | 用途 | 何时不需要 |
|------|------|----------|
| `session-log.md` | 每次任务摘要（7 天滚动）| 从不跳过 |
| `weekly-reviews.md` | 周复盘 | 一次性任务（选购决策完成后）|
| `archive/` | 历史归档 | 从不跳过 |

## 四、角色设计原则

**不预设角色，从两个问题推演：**

```
1. 这个领域最核心的「专业操作」有哪几类？
   → 每一类对应一个可能的专业角色

2. 这些操作能由同一个角色处理，还是需要拆开？
   → 拆开的标准：知识体系差异大 + 同时需要 + 协作价值高
   → 不要为了「看起来完整」而凑角色数量
```

**角色数量守则**（不变）：
- 2-3 个：简单协作，可不需要协调器
- 4-6 个：标准团队，需要协调器
- 7-10 个：考虑分组，子协调器
- **>10 个：不推荐**，拆分为多个独立团队

## 五、一个推演过程的示例

**用户输入**：「帮我备考 CFA 1 级，6 个月后考试」

**上帝的推演过程**：

```
Step 1: 识别领域 → 考试学习型

Step 2: 提设计问题
  Q: CFA 1 级有多少科目？
  A: 10 个模块（道德、数量、经济、财报、公司金融、权益、固收、衍生品、另类投资、组合管理）
  → 10 个模块，不可能 10 个教练，需要分组

  Q: 各模块权重一样吗？
  A: 不一样。道德模块约占 15%，财报+权益合计约 35%
  → 高权重模块（财报/权益）配专属教练，低权重模块可以合并

  Q: 6 个月，有时间节点吗？
  A: 考试是固定的，官方建议学习 300 小时
  → 需要进度追踪 + 阶段计划（前 3 个月打基础，后 3 个月刷题）

  Q: 是否有计算器考试要求？
  A: 是，必须用 BAII Plus
  → 需要计算器练习记录

  Q: 最担心什么？
  A: 道德卷（规则多，容易混淆）和数量（数学不好）
  → 这两个模块需要专属教练

Step 3: 推演出的团队结构
  协调器
  ├── 道德教练（Ethics）—— 规则记忆 + 场景判断
  ├── 数量/经济教练（Quant & Economics）—— 公式推导 + 计算器练习
  ├── 财报教练（FRA）—— 最高权重，精讲细练
  ├── 权益/固收教练（Equity & Fixed Income）—— 第二高权重
  └── 其他模块教练（衍生品/另类/组合/公司金融）—— 合并，效率优先

  shared/
  ├── user-profile.md（目标、基础、可用时间）
  ├── study-plan.md（300小时分配计划）
  ├── progress.md（模块进度 + 模拟题正确率）
  ├── weak-points.md（弱点状态机，精确到知识点）
  ├── mock-exam-log.md（模拟考成绩记录）
  ├── calculator-practice.md（BAII Plus 快捷键练习）
  ├── weekly-reviews.md
  ├── monthly-summary.md
  └── archive/

Step 4: 展示给用户确认
  「这是我推荐的结构，你觉得数量和经济合并成一个教练合理吗？
   另外，公司金融权重不算低（约 10%），要不要单独出来？」
```

**注意**：同样的「考试学习型」，雅思团队和 CFA 团队生成出来完全不同——角色数量不同、分组方式不同、记忆文件不同、复盘粒度不同。这才是上帝造物的本意。
