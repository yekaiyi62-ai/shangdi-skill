# Skill能力矩阵参考

> 设计Skill时，从这个矩阵中选择合适的能力组合。

## 一、输入类型

| 输入类型 | 说明 | 示例 |
|---------|------|------|
| **text** | 纯文本输入 | 一段作文、一个问题、一段代码 |
| **file** | 文件输入 | PDF文档、图片、音频、代码文件 |
| **url** | URL输入 | 网页链接、GitHub仓库 |
| **data** | 结构化数据 | JSON、CSV、表格 |
| **interactive** | 交互式多轮输入 | 口语对话、逐步引导 |
| **context** | 上下文环境 | 当前目录、git状态、项目结构 |

## 二、处理类型

| 处理类型 | 说明 | 适用场景 |
|---------|------|---------|
| **evaluate** | 评估/评分 | 作文评分、代码审查、简历评估 |
| **generate** | 生成内容 | 写文章、生成代码、创建模板 |
| **analyze** | 分析拆解 | 数据分析、竞品分析、需求分析 |
| **transform** | 格式转换 | 翻译、格式转换、重构 |
| **teach** | 教学辅导 | 出题、讲解、纠错、练习 |
| **plan** | 规划安排 | 学习计划、项目计划、时间表 |
| **diagnose** | 诊断排查 | Bug排查、问题诊断、健康检查 |
| **monitor** | 监控追踪 | 进度追踪、指标监控、状态报告 |

## 三、输出类型

| 输出类型 | 说明 | 格式要求 |
|---------|------|---------|
| **text** | 纯文本回复 | 适合简短回答和建议 |
| **report** | 结构化报告 | 必须有标题、摘要、正文、结论 |
| **score** | 评分结果 | 必须有分数、维度细分、改进建议 |
| **file** | 文件输出 | 明确文件格式和存储位置 |
| **structured_data** | 结构化数据 | JSON/表格/列表，有固定schema |
| **action_items** | 行动清单 | 可执行的下一步列表 |
| **diff** | 修改建议 | 原文对比，标注修改位置和原因 |

## 四、工具需求

| 工具 | 能力 | 什么时候需要 |
|------|------|------------|
| **WebSearch** | 联网搜索实时信息 | 需要最新数据、事实核查、行业动态 |
| **Read** | 读取文件 | 需要分析用户提供的文件 |
| **Write** | 创建/写入文件 | 需要输出文件（报告、代码等） |
| **Edit** | 编辑文件 | 需要修改现有文件 |
| **Bash** | 执行命令 | 需要运行脚本、编译、测试 |
| **Agent** | 启动子Agent | 需要并行处理、独立子任务 |
| **Glob/Grep** | 搜索文件 | 需要在代码库中查找信息 |

## 五、工作流模式

### 5.1 线性流程
```
输入 → Step 1 → Step 2 → Step 3 → 输出
```
适用：简单的转换、评估任务。

### 5.2 分支流程
```
输入 → 分类判断
         ├→ 类型A → 流程A → 输出A
         ├→ 类型B → 流程B → 输出B
         └→ 类型C → 流程C → 输出C
```
适用：需要根据输入类型走不同路径的任务。最常用。

### 5.3 循环流程
```
输入 → 处理 → 检查 → 不通过 → 修正 → 再检查 → 通过 → 输出
```
适用：迭代优化型任务（代码审查、写作修改）。

### 5.4 并行流程
```
输入 → ┬→ Agent 1 → 结果1 ┐
       ├→ Agent 2 → 结果2 ├→ 汇总 → 输出
       └→ Agent 3 → 结果3 ┘
```
适用：可以拆分为独立子任务的复杂分析。

### 5.5 团队流程
```
用户输入 → 协调器
             ├→ 分发给成员A → 结果A
             ├→ 分发给成员B → 结果B
             └→ 汇总/交接 → 综合输出
```
适用：多角色协同任务。

## 六、检查点设计

| 检查点类型 | 用途 | 设计原则 |
|-----------|------|---------|
| **确认检查点** | 关键决策前让用户确认 | 展示选项或方案，等待用户选择 |
| **展示检查点** | 中间结果让用户过目 | 展示摘要，用户可选择继续或调整 |
| **质量检查点** | 输出前自检 | 对照标准自动检查，标注通过/不通过 |

原则：
- 不要每一步都设检查点（太烦）
- 不可逆的操作前必须有检查点
- 第一次输出结果时必须有确认
- 检查点展示的信息必须简洁，不要信息过载

## 七、能力组合示例

### 示例1：代码审查Skill
```yaml
inputs: [context（当前diff）, file（代码文件）]
processing: [evaluate, diagnose]
outputs: [report（审查报告）, action_items（修改建议）]
tools: [Grep, Read, Bash（运行测试）]
workflow: 分支流程（按文件类型走不同审查标准）
checkpoints: [展示审查结果]
```

### 示例2：雅思写作教练
```yaml
inputs: [text（用户作文）, interactive（题目确认）]
processing: [evaluate, teach, generate]
outputs: [score（评分）, diff（修改建议）, text（范文）]
tools: [WebSearch（最新题目）, Write（保存练习记录）]
workflow: 线性流程（接收→评分→反馈→范文）
checkpoints: [展示评分后确认是否需要范文]
```

### 示例3：竞品分析团队
```yaml
# 成员1：数据采集员
inputs: [url（竞品网站）]
processing: [analyze]
outputs: [structured_data（竞品数据）]
tools: [WebSearch, Read]

# 成员2：分析师
inputs: [structured_data（来自成员1）]
processing: [analyze, evaluate]
outputs: [report（分析报告）]

# 协调器
routing: 用户给URL→成员1→成员2→输出报告
```
