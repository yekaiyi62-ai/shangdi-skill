# 团队Skill编排模式

> 多个Skill协同工作的设计模式参考。

## 一、团队架构模式

### 模式1：星型（总调度器）

```
          用户
           ↕
        协调器Skill
       ↙   ↓   ↘
    成员A  成员B  成员C
```

**适用场景**：成员之间独立工作，由协调器统一分发和汇总。
**示例**：雅思备考团队（用户找协调器，协调器分发给口语/写作/听力教练）。

协调器职责：
- 接收用户输入，判断应该转给哪个成员
- 维护用户档案和进度
- 汇总各成员的反馈，生成综合报告
- 协调学习计划和优先级

### 模式2：流水线

```
用户 → 成员A → 成员B → 成员C → 输出
```

**适用场景**：每个步骤的输出是下一步的输入。
**示例**：内容生产线（选题→写作→审核→排版→发布）。

### 模式3：并行独立

```
用户 → 成员A → 输出A
用户 → 成员B → 输出B
用户 → 成员C → 输出C
```

**适用场景**：每个成员完全独立工作，用户分别调用。
**示例**：公司职能团队（每个部门独立运作）。

### 模式4：混合模式

```
        用户
         ↕
      协调器
      ↙    ↘
   成员A   成员B → 成员C（B的输出给C）
```

**适用场景**：部分成员独立，部分有依赖关系。

## 二、知识库设计

> 详细的四层团队架构规范参见 `references/team-knowledge-architecture.md`。

### 三层架构概览

```
团队共享知识库 (references/)
  → 所有成员都可能用到的通用知识
  → 官方标准、领域总览、常见错误、来源索引
  → 包含 knowledge-index.md（知识路由表）

成员专属知识库 (members/[role]/references/)
  → 只属于某个角色的专业知识
  → 角色评分标准、题型分类、训练方法、反馈语料

用户记忆库 (shared/)
  → 动态变化的用户状态
  → 用户档案、进度记录、弱点画像、会话摘要
```

**核心原则**：
- 共享知识放团队级，专属知识放成员级，避免重复和标准不一致
- `shared/` 只放用户状态，不放静态知识
- 每个成员 SKILL.md 必须写明知识读取规则

### 知识索引 (knowledge-index.md)

每个团队必须在 `references/` 下生成 `knowledge-index.md`，作为知识路由表：

```markdown
| 任务类型 | 应读取文件 |
|---------|-----------|
| [任务1] | `references/xxx.md` + `members/[role]/references/xxx.md` |
| [任务2] | `references/xxx.md` + `shared/user-profile.md` |
```

协调器和成员根据此索引按需加载知识，不要一次性读取所有资料。

## 三、共享数据设计

### 用户档案 (user-profile.md)

团队成员共享的用户信息：

```markdown
# 用户档案

## 基本信息
- 目标：[用户的总目标]
- 当前水平：[评估结果]
- 时间规划：[截止日期、每天可用时间]

## 各维度评估
| 维度 | 当前水平 | 目标水平 | 最近评估日期 |
|------|---------|---------|------------|
| [维度1] | [分数] | [目标] | [日期] |
| [维度2] | [分数] | [目标] | [日期] |

## 学习偏好
- 偏好的学习方式：[视觉/听觉/实操]
- 薄弱环节：[列表]
- 已掌握：[列表]
```

### 进度追踪 (progress.md)

```markdown
# 学习进度

## 总进度
- 开始日期：[日期]
- 当前阶段：[阶段名]
- 总完成率：[百分比]

## 各模块进度
| 模块 | 已完成任务 | 总任务 | 最近活动 |
|------|----------|-------|---------|
| [模块1] | N | M | [日期：做了什么] |

## 练习记录
| 日期 | 模块 | 内容 | 得分 | 备注 |
|------|------|------|------|------|
| [日期] | [模块] | [练习内容] | [分数] | [改进建议] |
```

### 弱点画像 (weak-points.md)

```markdown
# 弱点画像

> 由各成员在练习中发现并更新，协调器汇总分析。

## 当前弱点

| 维度 | 弱点描述 | 严重程度 | 发现日期 | 来源 |
|------|---------|---------|---------|------|
| [维度] | [具体弱点] | 高/中/低 | [日期] | [哪个成员发现的] |

## 已解决的弱点

| 维度 | 弱点描述 | 解决日期 | 解决方式 |
|------|---------|---------|---------|
```

### 共享数据读写规则

1. **协调器有写权限**：负责创建和更新共享数据
2. **成员有读权限 + 追加权限**：可以读取共享数据，可以追加练习记录
3. **成员发现弱点时**：追加到 `shared/weak-points.md`，必须包含证据和来源
4. **冲突处理**：以最新时间戳为准
5. **数据位置**：统一存储在 `shared/` 目录下
6. **知识库文件不可日常修改**：`references/` 下的知识库文件在日常使用中不修改
7. **session-log 归档**：超过 7 天的记录移到 `archive/week-XX.md`

### 长期记忆规则（V2）

**每次任务完成后**：
1. 更新 `progress.md`（或领域对应的 log 文件）
2. 追加 `session-log.md`
3. 如发现新弱点 → 写入 `weak-points.md`，必须包含证据和来源
4. 如旧弱点连续 2 次通过 → 更新状态为"待复查"

**每周复盘**（协调器负责）：
1. 汇总本周 session-log
2. 写入 `weekly-reviews.md`
3. 将详细 session-log 归档到 `archive/week-XX.md`
4. 更新下周重点

**每月总结**（协调器负责）：
1. 汇总月度 weekly-reviews
2. 写入 `monthly-summary.md`
3. 重新评估目标和策略

**启动时按需读取**：
1. `shared/[主体]-profile.md` — 必读
2. `shared/weak-points.md` — 必读（如有）
3. `shared/progress.md` 最近 5-10 条 — 必读
4. `shared/weekly-reviews.md` 最近 2 周 — 推荐
5. `shared/archive/` — **不默认读取**，用户要求时才读

**领域动态结构**：
不同领域生成不同的 shared/ 文件。详见 `references/domain-patterns.md`。

## 四、协调器SKILL.md的核心结构

```markdown
## 问题路由

收到用户输入后，先判断应该转给哪个成员：

| 用户意图 | 转发给 | 触发词 |
|---------|--------|-------|
| [意图1] | 成员A | [关键词列表] |
| [意图2] | 成员B | [关键词列表] |
| [综合查询] | 协调器自己处理 | [关键词列表] |

## 转发规则

转发给成员时，必须附带：
1. 用户的原始输入
2. 用户档案摘要（从shared/user-profile.md读取）
3. 相关的历史记录（从shared/progress.md读取）

## 汇总报告

用户要求查看总进度时：
1. 读取 shared/progress.md
2. 读取各成员最近的练习记录
3. 生成综合报告，包含：各维度进度、薄弱环节、下阶段建议
```

## 五、成员Skill的团队适配

每个成员Skill除了自身的功能逻辑，还需要：

1. **知识读取规则**：明确写出每个场景该读哪些文件
2. **读取共享数据**：启动时自动读取 `shared/user-profile.md`，了解用户情况
3. **读取专属知识**：按需读取 `members/[self]/references/` 下的专属知识
4. **记录活动**：完成一次练习/评估后，追加记录到 `shared/progress.md`
5. **标准化输出**：输出格式统一，方便协调器汇总

```markdown
## 团队协作规则（在成员SKILL.md中）

### 知识读取规则

| 场景 | 读取文件 |
|------|---------|
| [场景1] | `members/[self]/references/xxx.md` + `references/official-standards.md` |
| [场景2] | `members/[self]/references/xxx.md` + `shared/progress.md` |

### 启动时
1. 读取 `shared/user-profile.md`，了解用户当前水平和目标
2. 读取 `shared/progress.md`，了解最近练习情况
3. 按需读取自己的 `references/` 下的专属知识

### 完成后
1. 将本次练习/评估结果追加到 `shared/progress.md`
2. 追加本次会话摘要到 `shared/session-log.md`
3. 如果发现用户水平有显著变化，更新 `shared/user-profile.md`
4. 如果发现新弱点，追加到 `shared/weak-points.md`，包含证据和来源
5. 如果旧弱点已通过复查，更新其状态
```

## 六、团队规模建议

| 团队规模 | 适用场景 | 注意事项 |
|---------|---------|---------|
| 2-3个Skill | 简单协作 | 可以不需要专门的协调器 |
| 4-6个Skill | 标准团队 | 需要协调器，推荐星型模式 |
| 7-10个Skill | 大型团队 | 考虑分组，每组一个子协调器 |
| >10个 | 不推荐 | 复杂度爆炸，拆分为多个独立团队 |
