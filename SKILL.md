# 上帝 · 造物术

> 造万物，不只是问答人偶——每个Skill都是有真实能力的专业工具。

## 核心理念

上帝造万物，三种Skill类型：
- **功能Skill**：有明确的输入→处理→输出流程，能调用工具，有专业工作流
- **人物Skill**：蒸馏人物思维框架，不仅能问答还有Agentic研究能力
- **团队Skill**：一次生成多个协同工作的Skill组，含总调度器

**核心特征**：
- 产物不只是聊天机器人，而是能做事的专业工具
- 支持一次生成协同工作的Skill团队
- 根据需求动态设计能力和工作流，不是固定模板

---

## 执行流程

### Phase 0: 需求解析

收到用户输入后，先判断要造什么：

#### Step 1: 需求分类

| 用户输入 | 类型 | 示例 |
|---------|------|------|
| 明确的单一功能需求 | **单Skill** → Phase 0A | 「做个代码审查助手」「造个翻译Skill」 |
| 需要多角色协作的需求 | **团队Skill** → Phase 0B | 「做个雅思备考团队」「生成产品团队」 |
| 明确的人名 | **人物Skill** → Phase 0C | 「蒸馏芒格」「做个费曼skill」 |
| 模糊的需求 | **诊断路径** → Phase 0D | 「我想提升写作能力」「帮我做个能管理项目的东西」 |

---

### Phase 0A: 单Skill需求确认

确认以下信息：

1. **这个Skill要解决什么问题？** 核心功能一句话描述
2. **输入是什么？** 用户会给它什么？（文本/文件/URL/数据/口头描述）
3. **输出是什么？** 用户期望得到什么？（评分/报告/修改建议/生成内容/文件）
4. **目标用户是谁？** 谁会用这个Skill？什么水平？
5. **需要联网吗？** 是否需要WebSearch获取实时信息？
6. **本地语料**：「你有没有这个领域的参考资料？方法论文档、评分标准、示例模板等——有的话直接给我，比我网上搜的精准。」

用户说「就做XX」没有更多信息 → 根据需求自动推断合理默认值，直接推进。

确认后 → Phase 1。

---

### Phase 0B: 团队Skill需求确认

用户需要一组协同工作的Skill。确认：

1. **团队要完成什么总目标？** 比如「雅思备考从6分到7分」
2. **需要哪些角色？** 列出初步角色设计（用户可增删改）
3. **角色之间如何协作？** 独立工作 vs 有依赖关系 vs 需要总调度
4. **需要共享数据吗？** 比如学习进度、用户档案
5. **平台**：Claude Code / Codex / 两者都要？

#### 角色设计展示格式

```
### 团队：[团队名称]
总目标：[一句话]

| # | 角色 | 核心能力 | 输入 | 输出 |
|---|------|---------|------|------|
| 1 | [角色名] | [能做什么] | [接收什么] | [产出什么] |
| 2 | ... | ... | ... | ... |

协调模式：[独立并行 / 流水线 / 总调度器]
共享数据：[有/无，具体是什么]
```

用户确认角色设计后 → Phase 1（对每个角色执行一次能力设计）。

---

### Phase 0C: 人物Skill需求确认

确认人名、聚焦方向、用途、本地语料。

确认功能模式：

| 模式 | 说明 |
|------|------|
| **标准模式** | 纯问答型人物Skill |
| **增强模式**（默认） | 除了问答，还有Agentic研究能力、工具调用、文件输出 |

确认后 → Phase 1。

---

### Phase 0D: 模糊需求诊断

用户不确定要什么，只有需求或困惑。通过1-2轮追问定位：

1. **你想让AI帮你做什么具体的事？** （不是"怎么想"，是"怎么做"）
2. **做完之后，理想的结果是什么样的？** （输出物是什么）
3. **这件事你现在是怎么做的？痛点在哪？** （定位自动化空间）

诊断后推荐2-3个方案（单Skill/团队/人物Skill），用户选择后进入对应Phase。

---

### Phase 1: 能力设计

**先设计能力再调研，不是拿到需求就直接搜索。**

对每个要生成的Skill，设计其能力矩阵：

#### 1.1 能力矩阵

```yaml
skill_name: [名称]
type: functional | person | hybrid

# 输入定义
inputs:
  - name: [输入名]
    type: text | file | url | data | interactive
    description: [描述]
    required: true | false

# 输出定义
outputs:
  - name: [输出名]
    type: text | file | report | score | structured_data
    format: [具体格式描述]

# 工具需求
tools:
  - WebSearch  # 需要联网搜索
  - Read/Write # 需要读写文件
  - Bash       # 需要执行命令
  - Agent      # 需要启动子Agent

# 工作流定义
workflow:
  - step: 1
    name: [步骤名]
    action: [做什么]
    checkpoint: true | false  # 是否需要用户确认
  - step: 2
    ...

# 领域知识需求
domain_knowledge:
  - [需要调研的领域知识1]
  - [需要调研的领域知识2]
```

#### 1.2 工作流设计原则

- **每个Skill必须有明确的问题路由**：收到输入后先分类，不同类型走不同处理流程
- **每个处理流程必须有具体的Step**：不是"分析一下"，而是"Step 1做什么，Step 2做什么"
- **关键步骤必须有检查点**：让用户确认后再继续
- **必须定义失败处理**：每个步骤如果失败了怎么办
- **输出必须有格式规范**：不是自由发挥，而是结构化输出

#### 1.3 团队协调设计（团队模式）

如果是团队Skill，额外设计：

```yaml
team:
  coordinator: [总调度Skill名称]
  shared_data:
    location: [共享数据目录]
    files:
      - user-profile.md    # 用户档案
      - progress.md        # 进度追踪
      - ...
  routing:
    - trigger: [什么时候调用哪个角色]
    - handoff: [角色之间如何交接]
```

**展示能力矩阵给用户确认** → 确认后进入Phase 2。

---

### Phase 2: 领域调研

根据Phase 1的 `domain_knowledge` 列表，执行定向调研。

#### 调研模式

| Skill类型 | 调研策略 |
|-----------|---------|
| **功能型** | 调研领域最佳实践、行业标准、评价体系、方法论 |
| **人物型** | 6维度调研（著作/对话/表达/他者/决策/时间线） |
| **团队型** | 每个角色分别调研，额外调研角色间协作模式 |

#### 功能型调研的Agent分配

根据领域知识需求，启动N个并行Agent（不固定为6个，按需求定）：

每个Agent的任务模板：
```
你的任务：调研[领域知识点]。

搜索方向：
- [具体搜索方向1]
- [具体搜索方向2]

输出要求：
- 写入 [skill目录]/references/research/[编号]-[主题].md
- 每条信息标注来源URL和可信度
- 提取可操作的规则、标准、流程，不要泛泛的概述
- 重点关注：评分标准、常见错误、最佳实践、行业规范

信息源黑名单：不使用知乎、微信公众号、百度百科。
```

#### 本地语料处理

用户提供的参考资料（评分标准PDF、方法论文档、示例模板等）优先级最高。先分析本地素材，网络调研只补充缺失部分。

#### 调研质量检查点

所有Agent完成后，展示调研摘要：

```
┌──────────────┬──────────┬──────────────────────────┐
│ 调研维度      │ 来源数量  │ 关键发现                  │
├──────────────┼──────────┼──────────────────────────┤
│ [维度1]      │ N篇      │ [关键发现]                │
│ [维度2]      │ N篇      │ [关键发现]                │
├──────────────┼──────────┼──────────────────────────┤
│ 信息不足维度  │ N个      │ [哪些维度]                │
└──────────────┴──────────┴──────────────────────────┘
```

用户确认 → Phase 3。

---

### Phase 3: Skill构建

将Phase 1的能力设计 + Phase 2的领域知识组装为可运行的Skill。

#### 3.1 读取模板

根据Skill类型选择模板：

| Skill类型 | 模板 |
|-----------|------|
| 功能型单Skill | `templates/functional-skill.md` |
| 人物型 | `templates/person-skill.md` |
| 团队协调器 | `templates/team-coordinator.md` |
| 团队成员 | `templates/functional-skill.md`（每个成员一份） |

#### 3.2 构建规则

**功能Skill的核心结构**：

```markdown
---
name: [skill-name]
description: |
  [功能描述、触发条件]
---

# [Skill名称]

## 能力范围
**能做的**：[列表]
**不能做的**：[列表]

## 问题路由
[输入分类表 → 不同场景走不同处理流程]

## 执行工作流
### 场景A: [场景名]
Step 1 → Step 2 → ... → 输出
### 场景B: [场景名]
...

## 领域知识
[从Phase 2调研填充的专业知识、评分标准、方法论]

## 输出格式规范
[每种输出的结构化格式定义]

## 质量检查
[自检清单]
```

**团队Skill的额外构建**：

1. 生成每个成员Skill的SKILL.md
2. 生成团队协调器的SKILL.md（负责分发任务、汇总结果）
3. 生成共享数据模板（user-profile.md等）
4. 生成团队README说明使用方式

#### 3.3 平台适配

构建完成后，根据用户选择的平台生成对应格式：

| 平台 | 输出位置 | 格式差异 |
|------|---------|---------|
| **Claude Code** | `.claude/skills/[name]/SKILL.md` | 标准SKILL.md + frontmatter |
| **Codex** | `.codex/agents/[name].md` 或 `AGENTS.md` 追加 | Codex agent格式 |
| **两者都要** | 同时生成两套 | |

Codex适配规则参见 `templates/codex-adapter.md`。

#### 3.4 目录结构

**单Skill**：
```
.claude/skills/[skill-name]/
├── SKILL.md              # 主文件
├── references/           # 领域知识
│   └── research/         # 调研原始数据
└── scripts/              # 辅助脚本（如需要）
```

**团队Skill**：
```
.claude/skills/[team-name]/
├── SKILL.md              # 团队协调器（入口）
├── members/
│   ├── [role-1]/SKILL.md
│   ├── [role-2]/SKILL.md
│   └── ...
├── shared/               # 团队共享数据模板
│   ├── user-profile.md
│   └── progress.md
├── references/           # 领域知识（团队共享）
│   └── research/
└── scripts/
```

### Phase 4: 质量验证

#### 4.1 功能测试

对每个Skill，用3个样本输入测试：

| 测试类型 | 方法 |
|---------|------|
| **正常路径** | 用典型输入走一遍完整工作流，检查输出格式和内容 |
| **边缘情况** | 用模糊/不完整的输入，检查错误处理和引导 |
| **输出质量** | 检查输出是否符合格式规范，是否有专业价值 |

#### 4.2 团队协同测试（团队模式）

- 协调器能正确分发任务到对应成员？
- 成员之间的数据交接正确？
- 共享数据的读写不冲突？

#### 4.3 通过标准

| 检查项 | 通过标准 |
|--------|---------|
| 问题路由 | 覆盖所有主要输入类型 |
| 工作流完整性 | 每个场景有完整的Step链 |
| 输出格式 | 有明确的格式规范，不是自由发挥 |
| 检查点 | 关键步骤有用户确认点 |
| 失败处理 | 每个步骤有失败后的处理方式 |
| 领域专业性 | 使用了专业术语和行业标准 |
| 能力边界 | 明确写了能做什么和不能做什么 |

验证通过 → Phase 5交付。不通过 → 标注问题，回到对应Phase修复。

---

### Phase 5: 交付

1. 将Skill文件写入对应平台的标准位置
2. 展示安装结果和使用方式
3. 用一个样本演示Skill的工作流程

#### 交付展示格式

```
✅ Skill生成完成

📁 文件结构：
[树形目录]

🚀 使用方式：
[具体的触发方式和示例输入]

📋 能力清单：
[这个Skill能做什么的简要列表]
```

## 特殊场景

### 更新已有Skill

用户说「更新XX」「给XX加个功能」时：

1. 读取现有SKILL.md，理解当前能力
2. 确认要更新什么（加功能/改流程/更新知识）
3. 只做增量修改，不重写整个Skill
4. 更新版本标注和时间戳

### 升级已有人物Skill

用户有纯问答型的人物Skill，想升级为功能Skill：

1. 读取现有人物Skill，保留心智模型和表达DNA
2. 追加能力矩阵（输入/输出/工具/工作流）
3. 升级Agentic Protocol为完整工作流
4. 输出增强版SKILL.md

### Codex兼容

生成Codex格式时的适配规则：
- 读取 `templates/codex-adapter.md` 获取格式转换规则
- Codex不支持frontmatter → 转为标题+描述段落
- Codex的agent目录结构不同 → 按Codex规范放置
- 工具调用语法可能不同 → 使用平台无关的描述

## 品味守则

| 原则 | 说明 |
|------|------|
| **能做事 > 会说话** | 一个能接收输入、处理、输出结果的Skill，比一个只会聊天的强100倍 |
| **具体 > 抽象** | 工作流的每个Step必须是可执行的具体操作，不是"分析一下" |
| **结构化 > 自由发挥** | 输出必须有格式规范，让用户知道会得到什么 |
| **诚实边界** | 明确写出做不到什么，比假装什么都能做更有价值 |
| **按需调研** | 不是所有Skill都需要深度调研，简单工具型Skill直接构建 |

### 绝不做的事
- 生成只有问答没有工作流的空壳Skill
- 把通用建议包装成"专业方法论"
- 生成没有输入输出定义的Skill
- 在信息不足时假装专业
- 一次生成超过10个Skill的团队（复杂度爆炸）
