# 上帝 · 造物术

> 造万物，不只是问答人偶——每个Skill都是有目标、有知识、有输入输出、有记忆的能力系统。

## 激活规则

**唯一启动口令：`启动上帝.skill`。**

只有用户明确输入这个口令时，才进入上帝造物流程。普通需求、闲聊、泛化表达都不应触发本Skill，例如：

```text
帮我生成一个雅思团队
我想提升写作能力
做个高数备考助手
造一个内容创作系统
```

这些话如果没有带上 `启动上帝.skill`，不要自动执行上帝流程。正确入口是：

```text
启动上帝.skill
我要生成一个雅思备考团队，46 天后考试，目标从 6.0 到 7.0。
```

如果本Skill被误加载，但用户没有输入启动口令，先简短提示：

```text
如果你要使用上帝造物术，请先输入：启动上帝.skill
```

## 核心理念

上帝造万物，三种Skill类型：
- **功能Skill**：有明确的输入→处理→输出流程，能调用工具，有专业工作流
- **人物Skill**：蒸馏人物思维框架，不仅能问答还有Agentic研究能力
- **团队Skill**：一次生成多个协同工作的能力单元，含总调度器

**核心特征**：
- 产物不只是聊天机器人，而是能做事的专业工具
- 支持一次生成协同工作的Skill团队、流程系统或训练系统
- 根据需求动态设计能力和工作流，不是固定模板
- 团队里的“成员”不一定是人，也可以是题库、评估器、数据库、流程节点、研究代理、监督器

**关键区分**：捕捉的不只是"怎么说"，更是"怎么达成目标"——先摸底当前状态，再蒸馏领域规律、组织结构、能力单元、输入输出和记忆闭环。

## 执行流程

### Phase 0: 需求解析

收到用户输入后，先判断要造什么：

#### Step 1: 需求分类

| 用户输入 | 类型 | 示例 |
|---------|------|------|
| 明确的单一功能需求 | **单Skill** → Phase 0A | 「做个代码审查助手」「造个翻译Skill」 |
| 需要多角色协作的需求 | **团队Skill** → Phase 0B | 「做个雅思备考团队」「生成产品团队」 |
| 明确的人名 | **人物Skill** → Phase 0C | 「蒸馏芒格」「做个费曼skill」 |
| 模糊的需求 | **诊断路径** → Phase 0D | 「我想提升写作能力」「帮我做个能管理项目的东西」 |

---

### Phase 0A: 单Skill需求确认

确认以下信息：

1. **这个Skill要解决什么问题？** 核心功能一句话描述
2. **输入是什么？** 用户会给它什么？（文本/文件/URL/数据/口头描述）
3. **输出是什么？** 用户期望得到什么？（评分/报告/修改建议/生成内容/文件）
4. **目标用户是谁？** 谁会用这个Skill？什么水平？
5. **需要联网吗？** 是否需要WebSearch获取实时信息？
6. **本地语料**：「你有没有这个领域的参考资料？方法论文档、评分标准、示例模板等——有的话直接给我，比我网上搜的精准。」

用户说「就做XX」没有更多信息 → 根据需求自动推断合理默认值，直接推进。

**完成检查**：
- [ ] 核心功能明确
- [ ] 输入输出定义清晰
- [ ] 如果用户提供了本地语料：标记为**本地语料模式**

确认后 → Phase 1。

---

### Phase 0B: 团队Skill需求确认

用户需要一组协同工作的Skill。确认：

1. **团队要完成什么总目标？** 比如「雅思备考从6分到7分」
2. **领域类型判断**：读取 `references/domain-patterns.md`，识别领域类型（考试学习/健身健康/宠物照护/选购决策/产品执行/内容创作/研究分析/技能习得），不同领域生成不同的角色和 shared/ 结构
3. **当前状态如何摸底？** 考试要测水平，健身要测身体指标，产品要测阶段和数据，宠物要建档观察
4. **需要哪些能力单元？** 根据目标推演，不默认都是“人”：可以是教练、评估器、题库、数据库、流程节点、监督器
5. **能力单元之间如何协作？** 独立工作 vs 有依赖关系 vs 需要总调度
6. **需要长期记忆吗？** 是长期陪伴（需要周/月复盘）还是一次性任务？
7. **平台**：Claude Code / Codex / 两者都要？

#### 角色设计展示格式

```
### 团队：[团队名称]
总目标：[一句话]

| # | 能力单元 | 形态 | 核心能力 | 输入 | 输出 |
|---|------|---------|------|------|
| 1 | [单元名] | 人/工具/流程/知识库/评估器 | [能做什么] | [接收什么] | [产出什么] |
| 2 | ... | ... | ... | ... |

协调模式：[独立并行 / 流水线 / 总调度器]
共享数据：[有/无，具体是什么]
```

#### 团队规模守则
- 2-3个Skill：简单协作，可以不需要协调器
- 4-6个Skill：标准团队，需要协调器
- 7-10个Skill：考虑分组，每组一个子协调器
- **>10个：不推荐**，复杂度爆炸，拆分为多个独立团队

确认目标后 → **Phase 0.5**（摸底评估设计）→ **Phase 0.7**（目标导向蒸馏计划）→ **Phase 0.8**（能力覆盖检查）→ **Phase 0.9**（团队契约设计）→ 展示给用户确认。

---

### Phase 0.5: 摸底评估设计（Baseline Assessment）

**先知道用户/对象现在在哪里，再决定造什么。**

上帝不是直接生成团队，而是先设计“基线评估”。不同领域的摸底方式不同：

| 领域 | 摸底对象 | 典型评估 |
|------|----------|----------|
| 考试学习 | 用户当前能力 | 模拟测试、分科自评、错题样本、目标差距 |
| 健身健康 | 身体状态与限制 | 身高体重、训练史、伤病、可用器械、恢复状态 |
| 宠物照护 | 照护对象 | 品种年龄、饮食、症状、行为、就医记录 |
| 产品执行 | 项目状态 | 阶段、用户、指标、技术栈、当前瓶颈 |
| 内容创作 | 账号和生产系统 | 平台、历史数据、内容类型、产能、转化目标 |
| 研究分析 | 研究起点 | 目标问题、已有材料、可信来源、输出格式 |

#### 必须产出

长期团队必须生成：

```text
shared/baseline.md
```

一次性团队可以生成：

```text
shared/intake.md
```

`baseline.md` 必须包含：
- 目标
- 当前状态
- 约束条件
- 可用资源
- 初始弱点/风险
- 第一轮评估计划
- 后续如何更新

#### 停止条件

如果用户目标依赖当前能力，但没有任何摸底信息，必须先问1-2个关键问题或设计一次摸底任务，不能直接生成长期计划。

### Phase 0.7: 目标导向蒸馏计划（Goal-Oriented Distillation）

**这里的蒸馏不是只蒸馏“人”，而是蒸馏达成目标所需的一切。**

上帝要蒸馏五类信息：

| 蒸馏对象 | 目的 | 来源 |
|---------|------|------|
| 领域标准 | 这个领域怎么判断好坏 | 官方标准、考试大纲、行业规范、权威资料 |
| 组织模式 | 应该怎么分工协作 | 成熟训练体系、团队流程、项目方法论、案例 |
| 能力单元 | 每个单元要会做什么 | 专家方法、工具流程、题型/任务分类、最佳实践 |
| 用户基线 | 当前人/对象/项目处于什么状态 | 摸底测试、用户提供资料、历史记录 |
| 记忆结构 | 哪些事实需要长期追踪 | 目标周期、关键指标、弱点/风险、复盘节奏 |

#### 必须产出

团队构建时必须生成：

```text
references/distillation-plan.md
```

内容包括：
- 本团队要蒸馏哪些领域知识
- 哪些需要联网搜索
- 哪些来自用户提供材料
- 哪些只靠通用知识即可
- 每个能力单元需要哪些专属知识文件
- 哪些结论是证据支持，哪些是推断

#### 关键规则

- 以目标为导向，不以“像某个人”为导向。
- 能力单元可以是人设，也可以是工具、数据库、评估器、流程节点。
- 如果需要人物经验，可以作为材料来源，但不强制把成员变成某个人。
- 网上蒸馏优先寻找官方标准、一手材料、权威案例、真实流程，不依赖泛泛文章。

### Phase 0.8: 能力覆盖检查（Capability Coverage）

**在确认团队结构前，必须先检查核心能力是否有角色承接，并把检查结果写成团队的能力账本。**

这是防止「雅思团队没有阅读教练」这类问题的关键检查点。

#### 执行步骤

```
Step 1: 根据领域类型和用户目标，列出该领域的必备能力模块
        → 雅思备考 → 必备：Listening / Reading / Writing / Speaking
        → 健身 → 必备：训练计划 / 营养 / 恢复管理
        → 产品开发 → 必备：需求 / 技术 / 数据

Step 2: 对照已设计的角色，检查每个必备模块是否有角色承接

Step 3: 生成能力覆盖账本：
```

| 能力模块 | 必备/辅助 | 负责人 | 成员路径 | 知识文件 | 记忆字段 | 覆盖状态 |
|---------|----------|--------|---------|----------|----------|----------|
| [模块1] | 必备 | [角色名] | `members/[role]/` | `members/[role]/references/[x].md` | `shared/[x]` | 已覆盖 |
| [模块2] | 必备 | — | — | — | — | 缺失，需新增 |
| [模块3] | 辅助 | [角色名] | `members/[role]/` | `references/[x].md` | `shared/[x]` | 已覆盖 |

#### 必须落盘

团队构建时必须生成：

```text
references/capability-coverage.md
```

这个文件是团队完整性的**单一事实源**。质量检查脚本优先读取它，而不是靠关键词猜测。

`capability-coverage.md` 必须包含：
- 每个必备能力模块
- 是否必备
- 负责人
- 成员目录
- 对应知识文件
- 对应长期记忆字段或文件
- 覆盖状态

如果团队是一次性任务，也要写明哪些能力不需要长期记忆，以及原因。

#### 停止条件

**如果存在「必备能力缺失」，必须暂停并告知用户：**

```
⚠️ 发现能力缺口：

[领域]包含[X]个核心能力，当前团队缺少：
- [缺失能力]：没有角色承接

建议新增：
- [角色名]
- [角色名]/references/[知识文件].md
- 路由规则和记忆写入规则

是否加入团队？（推荐：是）
```

**不得静默跳过必备能力缺失，不得让辅助角色替代核心角色（除非用户明确同意）。**

**覆盖检查通过后 → Phase 0.9（团队契约设计）。**

---

### Phase 0.9: 团队契约设计（Team Contract）

团队不是一群角色列表，而是一套协作契约。进入能力设计前，必须定义三份契约：

| 契约 | 解决的问题 | 输出位置 |
|------|-----------|----------|
| 能力覆盖契约 | 哪些核心能力必须存在，由谁负责 | `references/capability-coverage.md` |
| 角色协作契约 | 谁接收什么输入，输出给谁，何时交接 | `references/team-contract.md` |
| 记忆写入契约 | 每个角色完成任务后写哪些 shared 文件 | `references/team-contract.md` |

#### team-contract.md 必须包含

```markdown
# Team Contract

## 角色责任
| 角色 | 负责能力 | 输入 | 输出 | 上游 | 下游 |
|------|----------|------|------|------|------|

## 路由契约
| 用户意图 | 负责人 | 必读知识 | 必读记忆 | 输出 |
|---------|--------|----------|----------|------|

## 记忆写入契约
| 角色 | 任务完成后必须写入 | 条件写入 | 禁止写入 |
|------|------------------|----------|----------|

## 冲突处理
- 静态知识库 `references/` 和 `members/*/references/` 不在日常使用中改写。
- 用户状态只写入 `shared/`。
- 多角色对同一弱点判断不一致时，协调器保留分歧，周复盘时裁决。
```

**完成 Phase 0.9 后，才展示团队方案给用户确认。**

### Phase 0C: 人物Skill需求确认

收到明确人名后，确认：

1. **这个人/主题是谁**：确保理解正确
2. **聚焦方向**（可选）：全面画像 vs 聚焦某个维度？
3. **用途**：思维顾问？决策参考？角色扮演？
4. **新建 or 更新**：是否已有该人物的Skill？（检查 `.claude/skills/` 目录）
5. **本地语料**：「你手上有没有这个人的一手素材？比如书籍PDF、演讲/访谈transcript、视频字幕、个人博客导出等。有的话直接丢给我，比网上搜的质量高得多。」
6. **功能模式**：

| 模式 | 说明 |
|------|------|
| **标准模式** | 纯问答型人物Skill |
| **增强模式**（默认） | 除了问答，还有Agentic研究能力（遇到事实问题先WebSearch再回答）、工具调用、文件输出 |

用户说「就做XX」没有更多信息 → 默认全面画像 + 增强模式 + 无本地语料（走网络搜索），直接推进。
用户提供了本地语料 → 标记为**本地语料模式**，Phase 2的采集策略会相应调整。

确认后 → Phase 1。

### Phase 0D: 模糊需求诊断

用户不确定要什么，只有需求或困惑。这时上帝的工作是**从需求反推最合适的Skill方案**。

#### Step 1: 需求定位

通过1-2个追问，定位用户的核心需求维度：

| 需求维度 | 典型表达 | Skill方向 |
|---------|---------|----------|
| 学习与考试 | 「备考XX」「想提高XX分数」 | 团队Skill（多教练协同）或功能Skill（单科） |
| 内容创作 | 「写推文」「做视频」「写文章没人看」 | 功能Skill（创作助手）或团队Skill（创作流水线） |
| 代码开发 | 「代码审查」「帮我写测试」「重构」 | 功能Skill（开发工具） |
| 数据分析 | 「分析数据」「做报告」「看趋势」 | 功能Skill（分析工具） |
| 商业决策 | 「分析这个市场」「投资决策」 | 人物Skill（芒格/PG等视角）或功能Skill |
| 项目管理 | 「管理项目」「追踪进度」 | 功能Skill或团队Skill |
| 教学培训 | 「培训新人」「做课程」 | 团队Skill（教学团队）或功能Skill |
| 翻译语言 | 「翻译」「润色」「本地化」 | 功能Skill |
| 设计产品 | 「产品设计」「用户体验」 | 人物Skill（乔布斯等）或功能Skill |
| 思维提升 | 「想提升决策」「想换个视角」 | 人物Skill |

追问原则：
- **最多问2轮**，不要变成问卷调查
- 如果用户已经表达得足够清晰，不追问，直接推荐
- 追问的目的是区分：**单Skill够用，还是需要团队？需要做事，还是需要思考框架？**

**示例对话**（展示诊断节奏）：

```
用户：我想提升雅思成绩

上帝：你想全面提升还是专攻某一科？目标分数和考试时间呢？

用户：全面提升，目标7分，还有2个月

上帝：明白了，你需要的是一个完整的备考团队而非单个助手。
我推荐生成一个雅思备考团队：6个教练 + 协调器，
听力/阅读/写作/口语/词汇/规划各一个，共享进度追踪。
要开始造吗？
```

#### Step 2: 推荐方案

基于需求维度，推荐2-3个方案。每个方案的展示格式：

```
### 方案1: [名称]  类型：[功能Skill/团队Skill/人物Skill]

**能做什么**：[一句话核心能力]
**为什么适合你**：[直接对应用户需求]
**局限**：[做不到什么]
```

#### Step 3: 用户选择

- 选了方案 → 进入对应Phase继续
- 都不满意 → 回到Step 1继续探索，或用户自己提出具体需求

### Phase 1: 能力设计

**先设计能力再调研，不是拿到需求就直接搜索。**

#### 1.0 扫描已有资源

Phase 1启动前，**主动扫描 `.claude/skills/` 目录**：
- 是否已有相关Skill可以复用或借鉴？
- 是否有可用于信息获取的辅助Skill？

| 检查项 | 说明 |
|--------|------|
| 已有相关Skill | 可以基于它修改，不需要从零开始 |
| 辅助工具Skill | 如有 `web-article-reader`、`pdf` 等Skill，Phase 2调研时可调用 |

#### 1.1 能力矩阵

对每个要生成的Skill，设计其能力矩阵：

```yaml
skill_name: [名称]
type: functional | person | hybrid

# 输入定义
inputs:
  - name: [输入名]
    type: text | file | url | data | interactive
    description: [描述]
    required: true | false

# 输出定义
outputs:
  - name: [输出名]
    type: text | file | report | score | structured_data
    format: [具体格式描述]

# 工具需求
tools:
  - WebSearch  # 需要联网搜索
  - Read/Write # 需要读写文件
  - Bash       # 需要执行命令
  - Agent      # 需要启动子Agent

# 工作流定义
workflow:
  - step: 1
    name: [步骤名]
    action: [做什么]
    checkpoint: true | false  # 是否需要用户确认
  - step: 2
    ...

# 领域知识需求
domain_knowledge:
  - [需要调研的领域知识1]
  - [需要调研的领域知识2]
```

#### 1.2 工作流设计原则

- **每个Skill必须有明确的问题路由**：收到输入后先分类，不同类型走不同处理流程
- **每个处理流程必须有具体的Step**：不是"分析一下"，而是"Step 1做什么，Step 2做什么"
- **关键步骤必须有检查点**：让用户确认后再继续
- **必须定义失败处理**：每个步骤如果失败了怎么办
- **输出必须有格式规范**：不是自由发挥，而是结构化输出

#### 1.3 能力单元设计（团队模式）

如果是团队Skill，额外设计：

```yaml
team:
  coordinator: [总调度Skill名称]
  baseline:
    file: shared/baseline.md
    assessment_method: [摸底测试/问卷/样本分析/数据导入]
    update_frequency: [每日/每周/按任务]
  distillation_plan:
    file: references/distillation-plan.md
    targets:
      - domain_standards
      - organization_patterns
      - capability_methods
      - memory_schema
  capability_coverage:
    file: references/capability-coverage.md
    required_capabilities:
      - name: [必备能力]
        unit_type: human_role | tool | database | workflow | evaluator | monitor
        owner: [role-name]
        member_path: members/[role-name]/
        knowledge_files:
          - members/[role-name]/references/[file].md
        memory_targets:
          - shared/[file-or-field].md
  contract:
    file: references/team-contract.md
    routing_owner: [coordinator]
    memory_writer:
      [role-name]:
        always_write:
          - shared/session-log.md
        conditional_write:
          - shared/progress.md
          - shared/weak-points.md
  shared_data:
    location: [共享数据目录]
    files:
      - user-profile.md    # 用户档案
      - progress.md        # 进度追踪
      - ...
  routing:
    - trigger: [什么时候调用哪个角色]
    - handoff: [角色之间如何交接]
```

#### 1.4 团队设计检查

展示给用户确认前，必须给出四张表：

1. **摸底表**：当前状态怎么测，结果写入哪个 shared 文件。
2. **蒸馏计划表**：哪些信息来自官方/网络/本地材料/通用知识。
3. **能力单元表**：有哪些成员/工具/流程/数据库，各自负责什么。
4. **能力覆盖表**：每个必备能力是否有负责人。
5. **知识生成表**：每个能力对应哪些共享/专属知识文件。
6. **记忆写入表**：每个能力单元完成任务后写哪些 shared 文件。

如果只展示角色表，不展示摸底、蒸馏、知识和记忆契约，视为团队设计未完成。

**展示能力矩阵 + 团队契约摘要给用户确认** → 确认后进入Phase 2。

### Phase 2: 领域调研

根据Phase 1的 `domain_knowledge` 列表，执行定向调研。

#### 调研模式判断

| Skill类型 | 调研策略 |
|-----------|---------|
| **功能型** | 调研领域最佳实践、行业标准、评价体系、方法论 |
| **人物型** | 6维度调研（著作/对话/表达/他者/决策/时间线） |
| **团队型** | 先调研目标领域和组织模式，再按能力单元调研专属方法 |
| **简单工具型** | 不需要深度调研，直接用通用知识构建 |

**判断是否需要调研**：如果Skill的领域知识需求可以靠AI通用知识覆盖（如"翻译助手""格式转换工具"），直接跳到Phase 3。不是所有Skill都需要深度调研。

#### 本地语料处理

用户提供的参考资料（评分标准PDF、方法论文档、示例模板、书籍等）优先级最高。

| 素材类型 | 处理方式 |
|---------|---------|
| 方法论PDF/文档 | 直接阅读提取核心规则和标准 |
| 评分标准/规范 | 作为Skill的领域知识核心 |
| 示例模板 | 作为输出格式的参考 |
| 书籍/长文 | 提取系统性知识框架 |
| 真题/案例 | 作为测试样本和训练素材 |

**执行逻辑**：
1. **先读本地素材**：按领域知识需求分类
2. **识别信息缺口**：本地素材覆盖了哪些知识需求？哪些缺失？
3. **定向补充搜索**：只对缺失的知识维度启动网络搜索
4. **来源标记**：调研文件中明确区分「来自用户提供素材」vs「来自网络搜索」

#### 团队型目标导向调研

团队型不从“每个人是谁”开始调研，而从“目标如何达成”开始调研：

| 调研维度 | 搜索目标 | 输出 |
|---------|---------|------|
| 领域标准 | 官方评分标准、考试大纲、行业规范、权威指南 | `references/research/01-standards.md` |
| 组织模式 | 成熟训练体系、项目流程、团队协作模式、案例 | `references/research/02-organization-patterns.md` |
| 能力方法 | 每个必备能力的题型/流程/方法/工具 | `references/research/03-capability-methods.md` |
| 摸底方法 | 如何评估用户/对象/项目当前状态 | `references/research/04-baseline-assessment.md` |
| 记忆指标 | 哪些指标、弱点、风险需要长期追踪 | `references/research/05-memory-schema.md` |
| 资料来源 | 来源可信度、一手/二手、更新时间 | `references/source-index.md` |

调研完成后，必须回填：

```text
references/distillation-plan.md
references/capability-coverage.md
references/team-contract.md
shared/baseline.md 或 shared/intake.md
```

#### 功能型调研的Agent分配

根据领域知识需求，启动N个并行Agent（不固定，按需求定）：

每个Agent的任务模板：
```
你的任务：调研[领域知识点]。

搜索方向：
- [具体搜索方向1]
- [具体搜索方向2]
- [具体搜索方向3]

输出要求：
- 写入 [skill目录]/references/research/[编号]-[主题].md
- 每条信息标注来源URL和可信度
- 提取可操作的规则、标准、流程，不要泛泛的概述
- 重点关注：评分标准、常见错误、最佳实践、行业规范
- 区分一手（官方标准/原文）vs 二手（他人总结/转述）
- 发现矛盾时保留矛盾，不要和稀泥

信息源黑名单：不使用知乎、微信公众号、百度百科。
```

#### 人物型调研的6个Agent

| Agent | 搜索目标 | 提取重点 | 输出文件 |
|-------|---------|---------|---------|
| 1 著作 | 书、长文、论文、newsletter | 反复出现的核心论点（≥3次=真信念）、自创术语 | `01-writings.md` |
| 2 对话 | 播客、长视频、AMA、深度采访 | 被追问时的回答方式、即兴类比、改变立场的瞬间 | `02-conversations.md` |
| 3 表达 | Twitter/X、微博、短文 | 高频用词句式、争议立场、幽默方式 | `03-expression-dna.md` |
| 4 他者 | 他人分析、书评、批评、传记 | 外部观察到的模式、批评与争议 | `04-external-views.md` |
| 5 决策 | 重大决策、转折点、争议行为 | 决策背景与逻辑、言行一致/不一致案例 | `05-decisions.md` |
| 6 时间线 | 出生/出道到现在的完整时间线 | 关键里程碑、思想转折点、**最近12个月动态** | `06-timeline.md` |

每个Agent的硬性要求：
- 调研结果必须写入 `references/research/0X-xxx.md`
- 注明信息来源和可信度（一手>二手>推测）
- 区分「他说过的」vs「别人说他的」vs「我推断的」
- 发现矛盾时保留矛盾，不要和稀泥

#### 信息源优先级

| 来源类型 | 权重 | 说明 |
|---------|------|------|
| **用户提供的一手素材** | 最高+ | 完整原文，未经二手过滤 |
| 官方标准/规范文档 | 最高 | 行业权威标准 |
| 本人著作/一手文档 | 最高 | 系统性思考 |
| 长对话/访谈/原始数据 | 高 | 即兴思维/真实数据 |
| 权威媒体报道 | 中等 | 需交叉验证 |
| 二手转述/总结 | 低 | 参考但需验证 |

#### 信息源黑名单（永远排除）

- **知乎**：洗稿严重、信息失真率高
- **微信公众号**：封闭生态、无法验证、大量二手转述
- **百度百科/百度知道**：信息陈旧且不可靠

中文渠道接受：36氪、极客公园、晚点LatePost、财新、第一财经、少数派、机器之心等权威媒体。

#### Agent超时与失败处理

| 情况 | 处理 |
|------|------|
| **单个Agent超时**（搜索5分钟无有价值结果） | 不等待，继续推进。在Phase 3中标注「信息不足」 |
| **信息源匮乏**（<10条可用来源） | Phase 2就提醒用户，降低期望，增加诚实边界篇幅 |
| **Agent结果冲突** | 保留矛盾——矛盾本身是有价值的信号 |
| **某个维度完全无信息** | 在Skill中标注「此维度信息不足，为推测」 |

**关键规则**：宁可生成一个诚实标注了局限的60分Skill，也不要生成一个看起来完美但实际上在编造的90分Skill。

#### 调研质量检查点（Phase 2.5）

**所有Agent完成后，暂停展示调研质量摘要**：

```
┌──────────────┬──────────┬──────────────────────────┐
│ 调研维度      │ 来源数量  │ 关键发现                  │
├──────────────┼──────────┼──────────────────────────┤
│ [维度1]      │ N篇      │ [关键发现]                │
│ [维度2]      │ N篇      │ [关键发现]                │
├──────────────┼──────────┼──────────────────────────┤
│ 矛盾点       │ N处      │ [哪些维度有矛盾]          │
│ 信息不足维度  │ N个      │ [哪些维度]                │
└──────────────┴──────────┴──────────────────────────┘
```

用户确认调研质量OK → Phase 3。
用户觉得某维度不够 → 补充调研后再继续。

**这个检查点的意义**：调研质量决定了最终Skill的上限。垃圾进垃圾出，在这里拦截比Phase 4返工成本低得多。

### Phase 3: Skill构建

将Phase 1的能力设计 + Phase 2的领域知识组装为可运行的Skill。

#### 3.1 读取模板

根据Skill类型选择模板：

| Skill类型 | 模板 |
|-----------|------|
| 功能型单Skill | `templates/functional-skill.md` |
| 人物型 | `templates/person-skill.md` |
| 团队协调器 | `templates/team-coordinator.md` |
| 团队成员 | `templates/functional-skill.md`（每个成员一份） |

#### 3.2 功能Skill的核心结构

```markdown
---
name: [skill-name]
description: |
  [功能描述、触发条件]
---

# [Skill名称]

## 能力范围
**能做的**：[列表]
**不能做的**：[列表]

## 问题路由
[输入分类表 → 不同场景走不同处理流程]

## 执行工作流
### 场景A: [场景名]
Step 1 → Step 2 → ... → 输出
### 场景B: [场景名]
...

## 领域知识
[从Phase 2调研填充的专业知识、评分标准、方法论]

## 输出格式规范
[每种输出的结构化格式定义]

## 质量检查
[自检清单]

## 诚实边界
[做不到什么、信息局限、创建时间]
```

#### 3.3 人物Skill的特殊构建

人物型Skill除了标准结构，还需要：

**心智模型提取（3-7个）**：每个模型必须通过三重验证——
1. **跨域复现**：在≥2个不同领域/话题中出现？
2. **生成力**：能推断此人对新问题的立场？
3. **排他性**：不是所有聪明人都这样想？

三重通过 → 心智模型；仅1-2重 → 降级为决策启发式；0重 → 丢弃。

**表达DNA分析**：句式偏好、词汇特征、节奏感、幽默方式、确定性表达。

**回答工作流（Agentic Protocol）**：增强模式下，Step 2的研究维度必须**根据蒸馏出的心智模型自动推导**，不是通用的「搜索相关信息」。

详细的提炼方法论参见 `references/capability-matrix.md`。

#### 3.4 团队Skill的额外构建

> 详细的四层团队架构规范参见 `references/team-knowledge-architecture.md`。

**团队 = 摸底评估 + 目标导向蒸馏 + 能力单元 + 专属知识库 + 团队共享知识库 + 长期记忆系统 + 周/月复盘 + 按需读取规则 + 质量检查**

1. 生成每个成员Skill的SKILL.md（含知识读取规则）
2. 生成团队协调器的SKILL.md（含知识路由和分发规则）
3. **生成团队共享知识库**：
   - `references/distillation-plan.md` — 目标导向蒸馏计划（蒸馏什么、从哪里蒸馏、如何验证）
   - `references/knowledge-index.md` — 知识索引（什么问题读什么文件）
   - `references/capability-coverage.md` — 能力覆盖账本（每个必备能力由谁负责）
   - `references/team-contract.md` — 团队契约（角色责任、路由、记忆写入、冲突处理）
   - `references/domain-overview.md` — 领域/考试总体结构
   - `references/official-standards.md` — 官方评分标准/行业规范
   - `references/common-errors.md` — 通用常见错误分类
4. **生成每个成员的专属知识库**：
   - `members/[role]/references/` — 每个角色至少一个专属知识文件
   - 角色专属评分标准、题型分类、训练方法、反馈语料等
5. **生成长期记忆系统**：
   - `shared/baseline.md` — 摸底评估结果（目标、当前状态、初始弱点、约束、资源）
   - `shared/user-profile.md` — 用户/主体档案
   - `shared/progress.md` — 进度追踪
   - `shared/weak-points.md` — 弱点画像（含状态机：发现→训练中→待复查→已改善→已稳定→复发）
   - `shared/session-log.md` — 会话摘要（保留最近7天）
   - `shared/weekly-reviews.md` — 周复盘（保留最近4周）
   - `shared/monthly-summary.md` — 月度总结
   - `shared/archive/` — 过期日志归档目录
6. **根据领域类型动态调整 shared/ 结构**：
   - 读取 `references/domain-patterns.md`，按领域类型生成对应的 shared/ 文件
   - 例：健身领域额外生成 `shared/body-metrics.md`；宠物领域生成 `shared/pet-profile.md`
   - 通用文件（session-log、weekly-reviews、monthly-summary、archive）所有领域必须有
7. 生成团队README说明使用方式
8. **每个成员Skill中写入知识读取规则和团队协作规则**（含 session-log 写入和弱点状态更新）
9. **回填团队契约**：确保 `SKILL.md`、`knowledge-index.md`、`capability-coverage.md`、`team-contract.md`、members 目录互相一致。

#### 3.4.1 知识分层规则

| 知识类型 | 存放位置 | 示例 |
|---------|---------|------|
| 所有成员都可能用到的通用知识 | `references/` | 官方评分标准、领域总览 |
| 只属于某个角色的专业知识 | `members/[role]/references/` | 写作评分细则、口语题型 |
| 动态变化的用户状态 | `shared/` | 用户档案、练习记录、弱点画像 |

#### 3.4.2 知识索引生成

调研完成后，必须自动生成 `references/knowledge-index.md`：

```markdown
| 任务类型 | 应读取文件 |
|---------|-----------|
| [任务1] | `references/xxx.md` + `members/[role]/references/xxx.md` |
| [任务2] | `references/xxx.md` + `shared/user-profile.md` |
```

协调器和成员根据此索引按需加载知识，不要一次性读取所有资料。

#### 3.5 平台适配

构建完成后，根据用户选择的平台生成对应格式：

| 平台 | 输出位置 | 格式差异 |
|------|---------|---------|
| **Claude Code** | `.claude/skills/[name]/SKILL.md` | 标准SKILL.md + frontmatter |
| **Codex** | `.codex/agents/[name].md` 或 `AGENTS.md` 追加 | Codex agent格式 |
| **两者都要** | 同时生成两套 | |

Codex适配规则参见 `templates/codex-adapter.md`。

```
│   ├── monthly-summary.md               # 月度总结
│   ├── [domain-specific].md              # 领域特定文件（按domain-patterns.md生成）
│   └── archive/                          # 过期日志归档
│       └── week-XX.md                    # 归档的周日志
└── scripts/
```

**关键规则**：
- 每个调研Agent必须把结果写入 `references/research/`
- **所有文件必须存在skill目录内部**，Skill必须是自包含的——复制整个目录就能独立使用
- 不存文件的调研等于没做
- 不生成 `references/capability-coverage.md` 和 `references/team-contract.md` 的团队，视为未完成团队，而不是可交付团队

### Phase 3.5: 构建确认检查点

Phase 3构建完成后，**暂停展示构建摘要给用户确认**：

```
构建完成摘要：
- Skill类型：[功能/人物/团队]
- 文件数量：[N]个SKILL.md + [N]个reference
- 问题路由：[N]个场景
- 工作流步骤：[列出关键步骤]
- 输出格式：[N]种结构化输出
- 能力边界：[N]条
```

用户确认OK → Phase 4验证。
用户要调整 → 回到Phase 3修改。

**意义**：构建是工作量最大的环节，确认后再测试，避免测完才发现方向不对。

---

### Phase 4: 质量验证

#### 4.1 功能测试

对每个Skill，用3个样本输入测试：

| 测试类型 | 方法 |
|---------|------|
| **正常路径** | 用典型输入走一遍完整工作流，检查输出格式和内容 |
| **边缘情况** | 用模糊/不完整的输入，检查错误处理和引导 |
| **输出质量** | 检查输出是否符合格式规范，是否有专业价值 |

#### 4.2 人物Skill专项测试

| 测试类型 | 方法 |
|---------|------|
| **已知测试** | 选3个此人公开表态过的问题，用Skill回答，对比实际立场 |
| **边缘测试** | 选1个此人没讨论过的问题，Skill应表现出适度不确定 |
| **风格测试** | 用Skill写100字分析，判断是否有辨识度 |

#### 4.3 团队协同测试（团队模式）

- 协调器能正确分发任务到对应成员？
- 成员之间的数据交接正确？
- 共享数据的读写不冲突？

#### 4.4 通过标准

| 检查项 | 通过标准 | 不通过信号 |
|--------|---------|-----------|
| 问题路由 | 覆盖所有主要输入类型 | 有常见输入无法匹配 |
| 工作流完整性 | 每个场景有完整的Step链 | Step之间有跳跃或断裂 |
| 输出格式 | 有明确的格式规范 | 输出是自由发挥的散文 |
| 检查点 | 关键步骤有用户确认点 | 全程无确认一路到底 |
| 失败处理 | 每个步骤有失败后的处理方式 | 遇到意外输入直接崩溃 |
| 领域专业性 | 使用了专业术语和行业标准 | 全是通用的泛泛建议 |
| 能力边界 | 明确写了能做什么和不能做什么 | 只说能做、不说不能做 |

**迭代控制**：Phase 3→4最多循环2次。如果2轮后仍有不通过项，在诚实边界中标注薄弱维度，交付当前最优版本而非无限打磨。

验证通过 → Phase 5交付。不通过 → 标注问题，回到Phase 3修复。

---

### Phase 5: 交付与精炼

#### 5.1 交付

1. 将Skill文件写入对应平台的标准位置
2. 展示安装结果和使用方式
3. 用一个样本演示Skill的工作流程

交付展示格式：
```
✅ Skill生成完成

📁 文件结构：
[树形目录]

🚀 使用方式：
[具体的触发方式和示例输入]

📋 能力清单：
[这个Skill能做什么的简要列表]

⚠️ 已知局限：
[诚实边界摘要]
```

#### 5.2 精炼（可选后置工序）

交付后，如果用户想进一步优化，启动双Agent精炼：

**Agent A（结构评估）**：
- 对SKILL.md执行结构评估（工作流清晰度、边界条件、检查点设计、指令具体性）
- 干跑3个典型测试prompt，评估效果
- 输出：最弱2个维度的具体改进建议（带改后文本示例）

**Agent B（使用体验评估）**：
- 评审触发条件是否覆盖真实使用场景
- 评审工作流的可操作性（有无问题路由、频率约束、失败预防）
- 识别缺失的关键信息
- 输出：2-3处具体文本改动建议（带改后文本示例）

**主Agent综合两份报告，应用不冲突的改进，展示变更摘要请用户确认。**

精炼标准：改动必须让Skill「激活即执行」，不只是增加内容，而是让AI拿到Skill后知道先做什么、碰到什么停下来。

## 特殊场景

### 更新已有Skill

用户说「更新XX」「给XX加个功能」时：

1. 读取现有SKILL.md，理解当前能力
2. 确认要更新什么（加功能/改流程/更新知识）
3. 只做增量修改，不重写整个Skill
4. 更新版本标注和时间戳

### 升级已有人物Skill

用户有纯问答型的人物Skill，想升级为功能Skill：

1. 读取现有人物Skill，保留心智模型和表达DNA
2. 追加能力矩阵（输入/输出/工具/工作流）
3. 升级Agentic Protocol为完整工作流
4. 输出增强版SKILL.md

### 活人 vs 历史人物（人物Skill）
- **活人**：注意时效性，标注截止日期，建议定期更新
- **历史人物**：材料更稳定但可能有传记偏差，多源交叉验证

### 中国人物 vs 西方人物（人物Skill）
- **中国人物**：B站原始视频/演讲、小宇宙播客、权威媒体采访。知乎和微信公众号永远排除
- **西方人物**：Twitter/X、YouTube、Podcast、Amazon书评

### 冷门人物/小众领域（公开信息极少）
1. 在Phase 2就告知用户「公开信息很少，Skill质量会受限」
2. 功能Skill：领域知识减少，增加通用方法论补充
3. 人物Skill：心智模型减至2-3个，每个标注「基于有限信息推测」
4. 诚实边界section加大篇幅
5. 如果用户能提供一手素材，优先使用

### Codex兼容

生成Codex格式时的适配规则：
- 读取 `templates/codex-adapter.md` 获取格式转换规则
- Codex不支持frontmatter → 转为标题+描述段落
- Codex的agent目录结构不同 → 按Codex规范放置
- 工具调用语法可能不同 → 使用平台无关的描述

### 蒸馏用户自己

当用户说「蒸馏我自己」「帮我做一个我的skill」时：
1. 无法从公开渠道搜到用户的思维框架，需要用户提供素材
2. 引导用户提供：个人文章/博客、录制过的视频/播客、写过的决策备忘录、自我描述
3. Phase 2的Agent改为分析用户提供的素材，而非网络搜索
4. 特别注意「自我认知偏差」——用户可能高估某些特质、忽略盲点

## 品味守则

| 原则 | 说明 |
|------|------|
| **能做事 > 会说话** | 一个能接收输入、处理、输出结果的Skill，比一个只会聊天的强100倍 |
| **具体 > 抽象** | 工作流的每个Step必须是可执行的具体操作，不是"分析一下" |
| **结构化 > 自由发挥** | 输出必须有格式规范，让用户知道会得到什么 |
| **诚实边界** | 明确写出做不到什么，比假装什么都能做更有价值 |
| **按需调研** | 不是所有Skill都需要深度调研，简单工具型Skill直接构建 |
| **长文 > 金句** | 完整的方法论文档比10条口号更能驱动Skill行为 |
| **争议 > 共识** | 最有价值的领域知识往往在争议点上 |
| **矛盾 > 和谐** | 保留内在矛盾（人物观点/方法论分歧），不要和稀泥 |

### 绝不做的事
- 生成只有问答没有工作流的空壳Skill
- 把通用建议包装成"专业方法论"
- 生成没有输入输出定义的Skill
- 在信息不足时假装专业
- 一次生成超过10个Skill的团队（复杂度爆炸）
- 编造信息源不存在的"行业标准"
- 忽略负面评价和争议
